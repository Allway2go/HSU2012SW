주제 : HTML5 가 무엇인가, 왜 출현했는가, 현재상태와 미래상태 주요기능과 브러우저 지원 상태는?

1) HTML5란?
  HTML5는 HTML의 차기 주요 제안 버전으로 월드 와이드 웹의 핵심 마크업 언어이다. 2004년 6월 Web Hypertext Application Technology Working Group(WHATWG)에서 웹 애플리케이션 1.0이라는 이름으로 세부 명세 작업을 시작하였다. 2010년 3월 현재 WHATWG의 현재 초안 표준 상태(Draft Standard state)이며, W3C에서는 작업 초안 상태(Working Draft state)이다.
  HTML5는 HTML 4.01, XHTML 1.0, DOM Level2 HTML에 대한 차기 표준 제안이다. 이것은 어도비 플래시나 마이크로소프트의 실버라이트, 썬의 자바FX와 같은 플러그인 기반의 인터넷 애플리케이션에 대한 필요를 줄이는 데 목적을 두고 있다. (출처 : 위키백과사전)
  

  * WHATWG : W3C에서 1998년 HTML4.01을 발표하고 그들이 XHTML, XHTML2.0을 만들었으나 대부분의 브라우저 벤더들 (IE, OPERA, FireFox, Safari등)이 거부하면서 W3C를 이탕하기 시작하고 그에따라 만들어진 그룹
  * 마크업 언어 : hyper text markyp language의 약어로, 홈페이지에 보이는 것들을 동적으로 움직이는 것.
		  문서에 포함된 문장, 표, 소리 등과 같은 문서의 내용에 대한 정보가 아니라 그 문장과 그림, 표가 어떻게 배치되고 글자는 어떤 크기와 모양을 기지며, 들여쓰기와 줄간격, 여백은 어느 정도인지에 대한 정모를 의미.
  * DOM : 설계 최적화 모델 (Design Optimization Model), 문서 객체 모델 (Document Object Model)


2) 왜 출현했는가?
  월드 와이드 웹의 마크업 언어는 항상 HTML이었다. HTML은 원래 과학 문서를 의미에 따라 기술하기 위하여 디자인된 언어지만, 그 디자인과 이행은 몇 년에 걸쳐 변화되어 다른 타입의 문서를 기술하는데도 사용할 수 있게 되었다.
  웹 어플리케이션이라 불리는 모호한 분야는 HTML의 주된 분야인데도 적절히 다루어지지 않았다. 이 명세에서는 이것을 교정하는 시도를 할 것이며, 동시에 지난 몇 년간 제기되었던 이슈들을 다룰 수 있도록 HTML명세를 갱신할 것이다. 
  초창기의 5년간(1990-1995), HTML은 많은 개정을 거쳤고 많은 확장을 시도하였다. 이러한 시도는 처음에 CERN에서 이루어졌고, IETF로 넘어왔다
  W3C의 창설과 함께, HTML 개발의 중심은 다시 이동하였다. 수포로 돌아간 첫 번째 시도는 1995년에 있었는데, 이것은 HTML 3.0이라 한다. 좀 더 실용적인 접근이 1997년에 있었고 이것은 HTML 3.2이다. HTML4가 뒤를 이었고, 이것은 1998년에 완성되었다.
  이 시기에, W3C 회원들은 HTML의 진화를 멈추기로 하였고, 대신 XML에 기초한 대체품의 작업에 착수하였다. 이것이 XHTML이다. 이러한 노력은 HTML4를 XML로 재형성하는 작업으로 시작하였고, XHTML 1.0이라 알려졌다. XHTML1.0은 직렬화 이외에 새롭게 추가된 기능은 없었으며 2000년도에 완료되었다. XHTML 1.0이후, W3C의 초점은 다른 워킹 그룹들이 XHTML을 확장하는 작업을 쉽게 하는 것으로 바뀌었다. 이러한 노력을 XHTML 모듈화라 부른다. 이와 동시에, W3C는 기존의 HTML, XHTML 언어와 호환되지 않는 새로운 언어의 개발에 착수하였으며 이것을 XHTML2라고 칭했다.
  1998년 HTML이 그 진화를 멈추었을 무렵, HTML을 위한 API 일부가 브라우저 제작사들에 의해 개발되었고, DOM 레벨 1이라는 이름으로 명세화 되었다. 연이어 DOM 레벨 2 코어, DOM 레벨 2 HTML들이 개발되었다. 이러한 노력은 2004년에 몇 가지 DOM 레벨 3 명세들이 출판되면서 수그러들었지만, 그 워킹 그룹은 모든 레벨 3 명세들이 초안 단계에 이르기도 전에 종료되었다.
  2003년 XForms - 웹 폼의 새로운 세대가 될 기술 - 이 만들어지면서, HTML에 대한 대체품을 찾기보다 HTML 그 자체를 진화시키는 것에 관한 관심이 새롭게 고조되었다. 이러한 관심은 XML을 새로운 웹 기술로서 배포하는 것이 완벽히 새로운 기술에 제한되며 HTML처럼 이미 널리 알려진 기술에 대한 대체는 되지 못하였던 것에 기인한다.
  브라우저들이 이미 존재하는 HTML 웹 페이지와 호환되지 않는 렌더링 엔진을 개발할 필요 없이 XForms 1.0에서 소개된 기능들을 HTML4의 폼에 제공하는것이 가능하다는 증거가, 이 새롭게 고조된 관심의 첫 결과였다. 이러한 초기 시기, 초안들이 이미 공개되어 있었고 모든 곳에서 투입하길 갈망하고 있었지만, 명세의 저작권은 오직 오페라 소프트웨어의 소유였다.
  HTML의 진화를 다시 시작해야 한다는 아이디어는 2004년 W3C 워크샵에서 테스트 되었다. 이 테스트에는 HTML5작업의 기반이 되는 원칙들과 함께, 앞서 설명했던 폼 관련 기능들에 대한 초안이 모질라와 오페라에서 공동으로 W3C에 제안되었다. 이 제안은 거절되었다. W3C 스탭과 회원들은 XML에 기반을 둔 대체품을 웹의 진화를 위한 수단으로 이미 선택하였기 때문에, 새로운 제안과 충돌한다는 이유였다.
  그 직후, 애플, 모질라, 오페라는 공동으로 그들이 제안했던 내용을 계속해서 발전시켜 나갈 것을 선언하였다. 공개적인 메일링 리스트들이 만들어졌고, 그들의 초안은 WHATWG 사이트로 이동되었다. 이에 대한 수정, 그리고 그 명세를 재사용할 수 있는 저작권은 3개 회사가 공동으로 소유하게 되었다
  WHATWG는 몇 개의 핵심적인 원칙들에 기반을 둔다. 자세히 말해서 기술은 하위 호환선을 가져야 한다는 것, 명세와 이행은 서로 합치되어야 하며, 이행이 아니라 명세를 변경해서라도 이러한 합치를 이루어야 한다는 것, 그리고 명세는 충분히 자세하게 이루어짐으로써 이행안들이 서로 역컴파일하지 않고도 완전한 상호작용성을 가질 수 있어야 한다는 것이다.
  마지막 요구사항은 HTML5명세의 초점이 이미 3개로 분리되어 명시된 문서들 -HTML4, XHTML1, DOM2 HTML-을 포함하게끔 하였다. 또한, 기존에 일반적인 것으로 간주하던 것보다 훨씬 더 상세한 명세를 포함할 것이 요구되었다.
  2006년 W3C는 HTML5의 개발에 참가할 의사를 표명하였고, 2007년에 HTML5 명세의 개발을 위해 WHATWG와 함께 작업할 워킹그룹이 만들어졌다. 애플, 모질라, 오페라는 W3C가 관리하는 저작권을 통해 명세를 발행할 것을 허가하였는데, WHATWG사이트에 좀 덜 제한적인 라이센스를 제공하는 조건이였다
  그때부터 두 그룹은 함께 작업해오고 있다 (출처 : www.w3.org)

  결론적으로 왜 HTML5가 나왔냐면 여러 브라우저들 끼리 호환성이 높을 뿐만아니라 여러가지의 마크업과 그 마크업으로 인해 개발자들이 편하게 개발할 수 있는 환경을 제공해줌으로써 개발자들의 개발 시간도 줄어드는 효과를 얻을 수 있으므로 생산성이 더욱 높아지기 때문에 출현하였다고 본다.
  
  출처된 글을 읽고나서 생각해 보았더니 W3C라는 곳은 약간 박쥐같은 습성이 있는것 같다. 처음에 HTML을 개발하고 시장에 내 놓았고 버전을 업그레이드 하였지만 브라우저들 (크롬, 오페라, IE, FireFox, Safari 등)이 거부하면서 이런 브라우저들이 나와서 개발을 하여 선보인것이 지금의 HTML5의 처음만들어진 배경이 된다. 사람들이 많은 호응을 관심을 보이기 시작한 결과 W3C는 독립한 브라우저들을 무시하다가 현재에 이르러 독립된 브라우저 협회(WHATWG)와 손을 잡고 다시 HTML5라는것을 개발하고 있는 상황이다. 이를 비추어 봤을때 W3C는 좋은쪽으로만 붙으려는 경향이 보이는 생각이 들었다. 
  조사를 통해 또한가지 알아낸 사실이 있다. 바로 W3C에서 만든 HTML을 거부한 브라우저들중 IE 같은 경우 처음부터 다른 브라우저(크롬, 오페라 등)들과 HTML5를 개발한것은 아니었다는 사실이다. IE도 물론 W3C를 거부하였지만 그렇다고 HTML5를 사용하지 않았다. 하지만 현재 HTML5의 뛰어난 성능으로 인해 IE도 HTML5의 개발과 습득을 하고 있는 것은 사실이다. IE는 다른 브라우저들에 비해 늦게 HTML5라는 곳으로 뛰어들어서 IE10버전에 많은것을 HTML5로 맞추고 있지만 결국 다른 브라우저들에 비해 HTML5지원은 늦고 있다는것은 사실이다. 세계적으로 보았을때 아직 IE의 비율은 상당히 높은 편에 속한다. 이를 미루어 봤을때 IE도 다른 브라우저들과 같이 HTML5지원을 조속히 해야할 필요가 있다고 생각된다. 

  * CERN : 1954년에 스위스에 설립된 유럽 연합 공동 지원의 입자 물리학 연구소로 World Wide Web(월드 와이드 웹)의 개념이 처음 만들어진 곳이다. CERN의 컴퓨터 정보 수집 및 제어 관계 연구원이던 영국 출신의 팀 버너스 리는 1989년 인터넷을 편리하게 사용할 수 있는 하이퍼 텍스트(hypertext) 프로젝트를 제안했다.
  * IETF : 인터넷국제표준화기구 (Internet Engineering Task Force), 인터넷의 운영, 관리, 개발에 대해 협의하고 프로토콜과 구조적인 사안들을 분석하는 인터넷 표준화 작업기구로 망 설계자, 운영자, 업체, 연구자들에 의해 구성되었다.


3) 현재 상태는?
  현재 HTML5가 초안상태이기는 하지만 많은 브라우저 업체들이 이미 HTML5기능들을 지원하고 있다.
  구글 및 애플 닷컴 사이트의 첫 소스코드는 HTML5 Doctype을 이용하고 있다. 즉, 지금 웹 사이트에 HTML5 마크업을 이용할 수 있다. 물론 IE와 같이 오래된 브라우저의 경우 호환 스크립트로 기능 추가(Fallback)을 하는 방법을 사용하면 된다. HTML5 Shive, Modnizer, IE-CSS3 같은 스크립트를 통해 하위 호환성을 보장해 줄 수 있다. 단, HTML5를 통한 웹 사이트 제작이 회사의 개발 생산성을 높히고 고객의 요구와 비지니스에 부합한다는 전제하에 말이다. 미래 웹 기술을 미리 선도적으로 도입함으로서 부가 이익도 누릴 수 있다. 2004~2005년도에 웹 표준 기반 CSS 레이아웃 도입에 많은 토론이 있었지만 현재 결국 많이 도입되어 있다 (출처 : 실전 HTML5 가이드 pdf)

4) 미래 상태는?
  여러 브라우저들이 HTML5를 지향함에 따라 접할 기회가 많아지고 또 브라우저 뿐만 아니라 모바일까지도 HTML5의 영향력이 더운 커져서 새로운 웹 기술이기 때문에 학습 비용이 많이 들어도 HTML5의 기술을 배우는것이 필수가 될것이다. 또한 HTML5로 인한 업무 분담이 확실해 져서 개인의 영역이 분명해 지고 그 일에 매진할 수 있게 될 것이다.
  
5) 주요기능은?
  웹폼 (Web Form) : 사용자의 입력정보를 받기 위해 사용되는 입력 형태에 대한 정의에 사용되는 마크업.
  캔버스 (Canvas) : 웹에서 즉시모드(immediate mode)로 2차원 그래픽을 기리기 위한 API와 canvas 내 각종 객체를 회전, 변환하고 그레디언트, 이미지 생성 등 각종 효과를 주는 기능에 대한 API 지원
  SVG (Scalable Vector Graphic) : XML 기반의 2차원 벡터 그래픽을 표현하기 위한 언어
  Video/Audio : video는 비디오 또는 영화를 보여주기 위해 사용되는 미디어 엘리먼트이며, audio는 사운드나 오디오 스트림을 표현하기 위한 미디어 엘리먼트
  Geolocation : 디바이스의 지리적 위치 정보를 제공하는 API 표준 지원
  오프라인 웹 응용 (Offline Web Application) : 인터넷 연결이 지원되지 않는 경우에도 웹 응용이 정상적으로 수될 수 있도록 지원하는 기능으로 응용에 대한 캐싱과 데이터에 대한 캐싱으로 구성
  웹 SQL 데이터베이스 (Web SQL Database) : 다양한 표준 SQL을 사용해 질의할 수 있는 데이터베이스 기능에 대한 API 지원
  로컬저장소 (Local Storage) : 기존의 쿠키의 기능을 개선하기 위한 목적으로 개발된 기능으로 웹 클라이언트에서 키와 값이 쌍으로 구성된 데이터를 영구적으로 저장하는 기능
  웹 소캣 (Web Socket) : 웹 응용이 서버 측의 프로세스와 직접적인 양방향 통신을 위한 API 지원
  웹 워커 (Web worker) : 웹 응용을 위한 쓰레드(Thread) 기능에 대한 API 지원
  
6) 브라우저 지원상태는?
  2012년 4월 현재 HTML5를 완벽하게 지원하는 브라우저는 없다. 점차 HTML5를 지원하는 범위가 확대되고 있다.
